# 구축 계획서 — jw_pdf v2.0.0

> 작성일: 2026-04-07  
> 기능 정의서: [FEATURE_SPEC_v2.md](FEATURE_SPEC_v2.md)  
> 대상 플랫폼: Windows 11  
> 목표: v1.0.0 기반을 성능 중심으로 재설계하고, 신규 기능을 단계적으로 확장

## 1. 설계 원칙

### 1.1 핵심 원칙
| 원칙 | 설명 |
|------|------|
| **반응성 우선** | 모든 렌더링/변환/저장은 비동기. UI 스레드에서 100ms 이상 블로킹 금지 |
| **계층 분리** | core(순수 로직) / ui(PyQt6) / services(시스템 통합) 3계층. core에 PyQt6 import 금지 |
| **캐시 전략** | 렌더링 결과는 LRU 캐시. 편집 시 세대(generation) 카운터로 무효화 |
| **Graceful Degradation** | 외부 도구(Tesseract, LibreOffice, Pandoc) 미설치 시 해당 기능만 비활성화 |
| **점진적 로딩** | 대용량 PDF(1000+ 페이지)에서도 첫 페이지 1초 이내 표시 |

### 1.2 v1.0.0 대비 핵심 개선점

| 문제 (v1.0.0) | 원인 | 해결 (v2.0.0) |
|---------------|------|---------------|
| 줌/페이지 전환 시 UI 멈춤 | `_render_current()` 동기 렌더링 | QThreadPool 비동기 렌더링 엔진 |
| 같은 페이지 반복 재렌더링 | 캐싱 없음 | LRU 캐시 (줌 레벨×페이지×세대) |
| 편집 후 느림 | `reload_all()` + `goto_page()` 이중 렌더링 | dirty flag + 단일 지연 렌더 |
| 대량 페이지에서 썸네일 느림 | `processEvents` 루프 남용 | 가상화 (보이는 범위만 렌더링) |
| 앱 시작 느림 | 모든 모듈 즉시 로드 | Lazy import (OCR/서명/양식 등) |

## 2. 기술 스택

### 2.1 핵심 스택 (유지)
| 역할 | 라이브러리 | 비고 |
|------|-----------|------|
| GUI | PyQt6 >= 6.6.0 | |
| PDF 렌더링/어노테이션 | PyMuPDF (fitz) >= 1.24.0 | |
| PDF 페이지 편집 | pikepdf >= 8.0.0 | |
| 이미지 처리 | Pillow >= 10.0.0 | |
| PDF 생성 | reportlab >= 4.0.0 | |
| Markdown 변환 | markdown >= 3.5.0 | |
| 패키지 관리 | uv | |

### 2.2 신규 스택
| 역할 | 라이브러리 | 용도 |
|------|-----------|------|
| 디지털 서명 | pyHanko | PAdES B-B/B-T, PKCS#12 인증서 |
| OCR (1순위) | winocr | Windows 내장 OCR API (추가 설치 없음) |
| OCR (2순위) | pytesseract | Tesseract 설치 시 고정밀 OCR |
| 가상 프린터 | Ghostscript + Redmon | PostScript → PDF 변환 파이프라인 |
| 프린터 API | pywin32 (win32print) | 프린터 관리, 인쇄 기능 |
| 파일 감시 | watchdog | 가상 프린터 출력 파일 감지 |
| 설치 프로그램 | Inno Setup | exe 설치 + 레지스트리 + 프린터 드라이버 |

### 2.3 기술 검증 결과

| 항목 | 결론 |
|------|------|
| 가상 프린터 | Python 단독 불가. Ghostscript+Redmon 조합 또는 win32print+Windows Print-to-PDF 활용. 관리자 권한 필요. 구현 난이도 매우 높음 → 최후순위 |
| 파일 연결 | Windows 11은 레지스트리만으로 기본 앱 변경 불가. `winreg`로 ProgID 등록 후 `ms-settings:defaultapps` 안내. Inno Setup 설치 시 자동 등록 |
| OCR | winocr(Windows 내장)이 배포 용이. Tesseract는 70MB 번들 필요 → 선택 설치 |
| 디지털 서명 | pyHanko가 기능 완성도 최고. 한국 공동인증서(PKCS#12) 호환 |
| PDF 양식 | PyMuPDF widgets API로 읽기/쓰기/생성 모두 가능. 별도 라이브러리 불필요 |
| PDF 스레드 안전성 | fitz.Document는 스레드 안전하지 않음 → 워커 스레드별 별도 인스턴스 필수 |

### 3.2 핵심 컴포넌트 설계

#### (A) 비동기 렌더링 엔진 (`ui/render_engine.py`)

> v1.0.0 최대 병목인 동기 렌더링을 완전히 대체

```
                    ┌────────────────────────┐
                    │     PdfViewer (UI)      │
                    │  request_render(page,   │
                    │    zoom, priority)      │
                    └──────────┬─────────────┘
                               │ signal
                    ┌──────────▼─────────────┐
                    │    RenderEngine         │
                    │  ┌──────────────────┐   │
                    │  │ QThreadPool (4T)  │   │
                    │  │ ┌──────────────┐ │   │
                    │  │ │ RenderTask 1 │ │   │       ┌──────────────┐
                    │  │ │ RenderTask 2 │ │◄──┼───────│ LRU Cache    │
                    │  │ │ RenderTask 3 │ │   │       │ (page,zoom,  │
                    │  │ │ RenderTask 4 │ │───┼──────►│  generation) │
                    │  │ └──────────────┘ │   │       └──────────────┘
                    │  └──────────────────┘   │
                    └──────────┬─────────────┘
                               │ render_complete signal
                    ┌──────────▼─────────────┐
                    │  PdfViewer.on_rendered() │
                    │  → QPixmap 표시          │
                    └────────────────────────┘
```

**핵심 동작:**
- `request_render(page, zoom)` 호출 시 캐시 확인 → 히트면 즉시 반환
- 캐시 미스면 QRunnable 생성하여 QThreadPool에 제출
- 현재 페이지는 HIGH 우선순위, 프리페치(±3)는 LOW 우선순위
- 줌 변경 시: 기존 이미지를 스케일링하여 즉시 표시 → 백그라운드에서 정밀 렌더 → 교체
- 각 RenderTask는 독립 `fitz.Document` 인스턴스 사용 (스레드 안전)
- 페이지 편집/어노테이션 시 해당 페이지의 세대 카운터 증가 → 캐시 자동 무효화

#### (B) LRU 이미지 캐시 (`utils/cache.py`)

```python
# 캐시 키: (page_index, zoom_level_rounded, generation)
# 캐시 값: QPixmap
# 최대 용량: 20페이지 (약 200MB @ 150DPI A4)

#### (C) 이벤트 매개자 패턴

> v1.0.0은 21개 시그널 직접 연결 → 기능 추가 시 복잡도 폭발

v2.0.0에서는 MainWindow가 **매개자(Mediator)** 역할을 명확히 수행:

```
[PagePanel] ──page_selected──► [MainWindow(Mediator)] ──render_request──► [RenderEngine]
                                       │
[Toolbar] ──tool_changed──────►        ├──update_viewer──► [PdfViewer]
                                       ├──update_panel───► [PagePanel]
[SearchBar] ──search_text──────►       └──update_status──► [StatusBar]
```

모든 컴포넌트는 MainWindow에만 시그널을 보내고, MainWindow가 대상을 결정하여 전달.
컴포넌트 간 직접 참조 없음.

## 4. Phase 구현 절차 (TDD 기반) ⭐ 핵심 규칙

### 4.1 각 Phase 구현 시 반드시 아래 순서를 따른다

```
┌─────────────────────────────────────────────────────────────────┐
│  Step 1. 테스트 시나리오 작성                                     │

│          ✅ 성능 벤치마크 기준 충족 (해당 시)                      │
│          ✅ 에이전트 팀 검증 완료 서명                              │
│          → 통과해야만 다음 Phase 진행 가능                         │
├─────────────────────────────────────────────────────────────────┤
│  Step 7. 문서 업데이트 및 커밋                                    │
│          TEST_SCENARIOS.md, CHANGE.md, README.md 업데이트         │

### 4.2 에이전트 팀 역할 정의

각 Phase의 Step 1, 2, 5에서 에이전트 팀을 활용하여 품질을 보장한다.

| 에이전트 | 역할 | 투입 시점 |
|---------|------|----------|
| **시나리오 설계 에이전트** | 기능 정의서에서 TC/US 도출, 경계값·예외·오류 시나리오 발굴 | Step 1 |
| **교차 검증 에이전트** | 시나리오 설계 결과를 독립적으로 검토, 누락/모호/중복 지적 | Step 1 |
| **테스트 코드 검증 에이전트** | 작성된 테스트 코드가 시나리오를 정확히 반영하는지 확인 | Step 2 |
| **비기능 검증 에이전트** | 성능 벤치마크, 보안 취약점, 메모리 누수, 스레드 안전성 관점 검증 | Step 5 |

> **참고**: Phase 1~2(핵심)에는 Step 1/2/5 모두에서 에이전트 검증을 수행한다. Phase 3 이후에는 Step 5 최종 검증만 수행하여 오버헤드를 줄인다.

**에이전트 검증 프로토콜:**
```
1. 최소 2개 에이전트가 병렬로 독립 검증 수행
2. 각 에이전트는 발견한 문제를 [심각/보통/경미]로 분류
3. [심각] 이슈가 1개라도 있으면 해당 Step 재수행
4. [보통] 이슈는 수정 후 진행, [경미]는 기록 후 진행 가능
5. 검증 통과 시 커밋 메시지에 "[Phase N Gate PASS]" 태그 기록
```

### 4.4 Phase 의존 관계

```
Phase 1 (성능)  ─────► Phase 2 (뷰어) ─────► Phase 3 (페이지 도구)
                                │                      │
                                ▼                      ▼
                          Phase 4 (어노테이션)    Phase 5 (보안/양식)
                                │
                                ▼
                          Phase 6 (고급 도구)
                                │
                                ▼
                          Phase 7 (시스템 통합) ─► Phase 8 (가상 프린터)
```

| Phase | 선행 의존 | 이유 |
|-------|----------|------|
| Phase 2 | Phase 1 | 비동기 렌더 엔진 필요 |
| Phase 3 | Phase 1 | 렌더 캐시/세대 카운터 필요 |
| Phase 4 | Phase 2 | 텍스트 선택 기능 필요 (하이라이트/밑줄의 전제) |
| Phase 5 | Phase 1 | 기본 렌더 인프라 필요 |
| Phase 6 | Phase 2 + 4 | 검색 + 어노테이션 기반 기능 |
| Phase 7 | Phase 2 | 설정 시스템 필요 |
| Phase 8 | Phase 7 | 설치 프로그램 필요 |

> Phase 3과 Phase 5는 서로 독립이므로 병렬 진행 가능.

### 4.5 Phase 게이트 체크리스트 (공통)

모든 Phase는 아래 조건을 **전부 충족**해야 다음 Phase로 진행한다:

```
□ 해당 Phase의 신규 TC 전체 PASS
□ 해당 Phase의 성능 벤치마크 충족 (해당 시)
□ 이전 모든 Phase의 TC + 벤치마크 전체 PASS (회귀 테스트)
□ v1.0.0 기존 테스트 262개 전체 PASS (회귀 테스트)
□ 에이전트 팀 최종 검증 완료 → 커밋 메시지에 "[Phase N Gate PASS]"
□ TEST_SCENARIOS.md에 해당 Phase 시나리오 추가 완료
□ CHANGE.md 업데이트 완료
□ 코드에 TODO/FIXME 없음 (해당 Phase 범위 내)
```

> **벤치마크 정책**: 성능 벤치마크는 Phase 1에서 5개 작성하며, 이후 모든 Phase의 회귀 테스트에 포함된다. 게이트 카운트에도 벤치마크를 포함한다.

## 8. 위험 관리

| 위험 | 심각도 | 완화 방안 |
|------|--------|----------|
| fitz.Document 스레드 안전성 | 높음 | 워커 스레드별 독립 인스턴스 + 파일 잠금 관리 |
| 가상 프린터 드라이버 서명 | 높음 | Phase 8로 후순위. 대안: Windows Print-to-PDF + watchdog 파일 감시 |
| pyHanko 유지보수 중단 | 중간 | 버전 고정, 서명 기능은 Phase 6으로 후순위 |
| 캐시 메모리 폭발 | 중간 | 용량 제한(200MB), 거리 기반 우선 제거 |
| Windows 11 기본 앱 변경 제한 | 중간 | 프로그래밍적 변경 불가 → 설정 페이지 안내 + Inno Setup 등록 |
| Tesseract 번들 크기 (70MB+) | 낮음 | winocr 우선, Tesseract는 선택 설치 |
| 한국 공동인증서 확장 필드 | 낮음 | pyHanko PKCS#12 호환 확인, 미지원 시 안내 |
